# Component UI tests (Playwright) — proposed workflow

This document describes end-to-end testing of the library's components against the **Qwik City demo** app. **Playwright** lives at the repository root (`playwright.config.ts`, the `e2e/` folder). Below is the workflow and conventions; add further components as `e2e/ui/base/<slug>.spec.ts`.

**Playwright runs from the repository root** (`playwright.config.*` sits next to `demo/`, `components/`). The `webServer` in the config starts the dev server from `demo/` (e.g. `npm run dev` with `--host` and a fixed port); `baseURL` points at that server.

## Goal

- Verify **functionality** (clicking, keyboard, form state, opening dialogs, etc.) on a real page.
- Verify **appearance** (layout regressions) using screenshots or Playwright's built-in snapshots (`expect(page).toHaveScreenshot()`), optionally comparing only critical states visually.

## Context: demo and QUI

- The demo app typically lives in the `demo/` folder with `qui.config.json` (`targetPath`, a connected component source).
- The pages for individual components are generated by the **`qui generate-demo`** command (the default route base is `/qui-demo`, changeable via `--route-base`).
- The URL of a single component has the form:

  `/<route-base>/components/<uilib>/<slug>/`

  Example for the default library **`base`** and the **dialog** component:

  `/qui-demo/components/base/dialog/`

- The demo config often features a single UI library (e.g. `base` in `qui.config.json` → `uilibs`). Tests can take the **default `uilib` = `base`** and parameterize it (environment variable, constant in `playwright.config`) so additional libraries can be added later without rewriting paths.

## Steps before testing

### 1. Have an up-to-date demo with the components copied in

**New demo project** (from the `q-ui-lib` root there is a shortcut):

```bash
npm run qui:createdemo
```

or manually: `qui init demo`, in `demo/` run `qui add …`, then `qui generate-demo` per the CLI docs.

**Existing `demo/`** — sync the contents of installed components with their source and regenerate the demo routes as needed:

```bash
cd demo
node ../bin/qui.js update --all
node ../bin/qui.js generate-demo
```

- **`update --all`** — overwrites all already-installed components in `targetPath` (e.g. `src/components/ui`) from the connected repository.
- **`generate-demo`** — recreates/updates the routes under `src/routes/<route-base>/components/…` based on what is installed in `targetPath`. Run it again after adding new components or changing the demo route structure.

> Note: `update --all` takes its "seeds" from **all** installed components in `targetPath`, not from a single `uilib` argument. The library selection (`base` vs another) is driven by the contents of the folders under `targetPath` and `qui.config.json`.

### 2. Start the demo server for Playwright

In the root `playwright.config`, use `webServer`: the command from `demo/` (working directory `demo`), a fixed port, and `reuseExistingServer` for local development. Tests then use `baseURL` (e.g. `http://127.0.0.1:5173`).

## Test directory structure (proposed)

Repository root:

```text
q-ui-lib/
├── playwright.config.ts          # or .mts; baseURL, webServer → demo
├── package.json                  # scripts: playwright test, install browsers
├── demo/                         # Qwik app (test target)
├── e2e/
│   ├── support/
│   │   ├── demo-routes.ts        # building paths: /qui-demo/components/{uilib}/{slug}/
│   │   └── constants.ts          # ROUTE_BASE, DEFAULT_UI_LIB ('base'), baseURL helper
│   └── ui/
│       ├── base/                 # segment = uilib in the URL; default library
│       │   ├── dialog.spec.ts
│       │   ├── tooltip.spec.ts
│       │   └── …                 # one *.spec.ts per leaf component (slug = file name)
│       └── <other-uilib>/          # optionally more libraries using the same pattern
├── test-results/                 # run outputs (gitignored)
└── playwright-report/            # HTML report (gitignored)
```

*(Name leaf components inside nested slugs, e.g. `menu-item`, as `menu-item.spec.ts`; optionally a subfolder `e2e/ui/base/menu/` only if the demo has multiple routes under one "name" — the default is the flat `e2e/ui/<uilib>/<slug>.spec.ts`.)*

**Conventions**

| Path on disk | Maps to |
|----------------|-----------|
| `e2e/ui/<uilib>/<slug>.spec.ts` | Demo URL `/<route-base>/components/<uilib>/<slug>/` |
| `e2e/support/` | Shared imports (URLs, testID maps, waiting for hydration) — **not** `*.spec.ts` |

**Visual regression:** Playwright stores snapshots next to the spec in a `<name>.spec.ts-snapshots/` folder (you can keep the default; optionally centralize via `snapshotPathTemplate` in the config if you need a single folder for all snapshots).

**Alternative:** instead of a top-level `e2e/`, use `tests/e2e/…` with the same internal `ui/<uilib>/` hierarchy — functionally identical, just a longer path. The recommendation here is **`e2e/`** for clarity next to `demo/`.

## Running Playwright tests "per component"

### Functionality

- For each component (or for a group of related scenarios) use one file or a `test.describe` named after the component **slug** (e.g. `dialog.spec.ts`).
- The test navigates to:

  `{baseURL}/qui-demo/components/{uilib}/{slug}/`

  where the **default `uilib` = `base`** unless overridden.
- Assertions: visibility, ARIA roles, text after interaction, element counts, URL query (if the demo uses it), etc.

### Appearance

- After the layout stabilizes (e.g. `await expect(locator).toBeVisible()` and possibly `page.waitForTimeout` only where you can't wait for a specific state), take a screenshot of the whole page or of the key container.
- For components with animations, disable animations in the test build or via `reduced-motion` / test-only CSS so screenshots are deterministic.

### Running for a single component / single library only

- From the root: `npx playwright test e2e/ui/base/dialog.spec.ts` or `npx playwright test -g "dialog"`.
- Parameterizing `uilib`: the `e2e/ui/<uilib>/` folder already selects the library; for shared helpers, optionally set `UI_LIB=base` in CI if some specs are generic.

## Summary of steps

1. **Prepare the demo:** either generate a new demo, or in an existing `demo/` run `qui update --all` and `qui generate-demo` as needed.
2. **Set up Playwright at the root** (`playwright.config.*`, dependencies in the root `package.json`), `webServer` → `demo/`, tests in `e2e/`.
3. **For each component** create `e2e/ui/base/<slug>.spec.ts`, navigate to the demo URL, run assertions and optionally screenshots.
4. **Default `uilib`** in the proposed conventions: **`base`** (folder `e2e/ui/base/`).

---

**Root scripts:** `npm run test:e2e` (starts the dev server in `demo/` on port **5173** and runs the tests), `npm run test:e2e:ui`, and on a clean machine first `npm run playwright:install` (browsers). Config: `playwright.config.ts` (`webServer`, `baseURL`). Add CI steps as needed (e.g. `cd demo && npm ci` before `test:e2e`).
